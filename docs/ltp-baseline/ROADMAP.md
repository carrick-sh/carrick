# LTP DIFF fix-roadmap (ltp-diff-triage workflow, 24 agents, 293 DIFFs → 133 clusters)

_The three biggest levers are all high-confidence, mechanical, many-test wins: (1) a shared path-resolution errno helper (ENAMETOOLONG/ELOOP/intermediate-ENOTDIR) unblocks ~24 fs tests plus pathconf02/utime07 with one piece of infrastructure; (2) the Linux set*id credential model in creds.rs (saved-id rule + non-root EPERM) clears ~20 process tests and is the keystone the privilege-EPERM clusters across fs/ipc/mm/other depend on; (3) the high-VA file/shm alias teardown fix is the single highest-leverage mm change (7 tests, plus it resolves the two needs-trace mprotect failures). After those, a cluster of cheap missing-syscall/errno wins (ioprio, vhangup, signalfd4, sched negative-pid EINVAL, pidfd CLOEXEC, getrandom/capget/fadvise/ftruncate errno, mkdir/chmod setgid, flock/removexattr host-forwards) each clear tests at near-zero effort. host-limitation (SCTP, sparse lseek, pipe capacity, large-file sendfile, RLIMIT_CPU), framework-setup (kconfig/proc-sysvipc), and needs-carrick-trace clusters are demoted to the tail._

## Adversarial verdicts (top 12)

**#1 Shared path-resolution errno helper (ENAMETOOLONG / ELOOP / intermediate ENOTDIR)** — SOUND: Confirm: genuine under-enforcement, not jitter. resolve_at_path (fs.rs:1116) has no length/component/loop checks; resolve_following returns None on loop (fs_backend.rs:713) which rootfs.lookup collapses to ENOENT (rootfs.rs:503), so ELOOP/ENAMETOOLONG/intermediate-ENOTDIR are never synthesized. Pure errno synthesis — no missing macOS primitive — and dispatcher-level component-walking is proven feasible (dac_ancestors_searchable). All baseline diffs are clean (passed,failed) divergences, no inversions/broken/conf. Cluster holds: utime07 and pathconf02 (name=0=_PC_LINK_MAX → glibc __statfs(path) → carrick statfs handler, same resolve_at_path+layered_metadata gap) genuinely share the resolver gap. Effort "large" is right (~20 call-sites, ~14 handlers). One nit: list names "pathconf" but the real entry point is statfs/statfs64 — covered by the "route stat*" intent, not disqualifying.

**#2 Linux set*id credential model (saved-id rule + non-root EPERM)** — SOUND: CONFIRM (sound): real bug, not jitter. Current creds.rs accepts all set*id and never updates suid/sgid; the fix's saved-id rule reproduces setreuid02's deterministic root-only table exactly (incl. the subtle "euid==old ruid → suid unchanged" case), setresuid is correctly kept explicit-only (matches setresuid01's 9 success rows + man page), EPERM matches the {r,e,s} allowance, and the apt _apt privsep drop + existing concurrency test still pass (privileged at call time). Pure in-runtime emulation — no macOS primitive needed. Two caveats, neither blocks actionability: (1) the cluster description mis-attributes setregid02 to the saved-id rule — that test actually drops to nobody and expects EPERM, never verifying sgid, but the fix's EPERM half still makes it pass; (2) effort "medium/20" is slightly generous for a 6-handler single-file logic edit but defensible given the fiddly per-syscall rule matrix.

**#3 High-VA file/shm alias teardown + stage-1 invalidation** — FLAW: Mostly sound + actionable for the genuine alias cluster (munmap01/mprotect02/shmdt01/mremap04: hv_vm_unmap + hv_vm_protect DO exist in applevisor-sys-1.0.0, stage-1 invalidate/protect_range already work on alias L2 blocks, and the LTP tests are single-vCPU so the documented arm64 stage-2-TLB wall doesn't bite). But mprotect04 is MISCLASSIFIED into this cluster: it uses MAP_PRIVATE|MAP_ANONYMOUS arena maps + a PROT_EXEC function-pointer call under the legacy framework, never touches the high-VA alias range, so its 0/0 will not be fixed by alias teardown. The proposal explicitly claims mprotect04's 0/0 "is this same defect" — that is the flaw.

**#4 signalfd4 (syscall 74) implementation** — SOUND: Confirm: genuinely actionable. macOS has no signalfd, so missing-syscall/emulate (not host-forward) is correctly classified; syscall 74 is a real BootstrapStub ENOSYS today, not Docker jitter. All named hooks verified to exist: flag validator at mod.rs:1870, take_deliverable_pending/mark_signal_pending in signal.rs, and the TimerFd/EventFd OpenDescription + install_fd + linux_fd_flags_from_open_flags + OpenDescriptionBase(status_flags)/fd_flags pattern that F_GETFD/F_GETFL read back; registering 74 => signalfd4 mirrors 19 => eventfd2. Two non-fatal caveats: (1) the actual LTP cluster is only signalfd4_01 (SFD_CLOEXEC) + signalfd4_02 (SFD_NONBLOCK), both pure fcntl flag checks — neither read()s signals, so the read/siginfo/poll-epoll machinery is gold-plating beyond the tests (a flag-only stub would pass all assertions); (2) the read path can't reuse take_deliverable_pending verbatim — signalfd consumes BLOCKED masked signals, but that helper returns NON-blocked ones (take_pending_in with !mask), so the correct filter is the opposite, an implementation detail not exercised by the tests. Effort "medium" is fair-to-generous.

**#5 openat2 validation + RESOLVE_* flags + size handling** — FLAW: Size-validation half is sound, but the RESOLVE_* half is wrong: mapping NO_SYMLINKS/NO_MAGICLINKS->ELOOP and NO_XDEV/BENEATH->EXDEV unconditionally breaks LTP openat202's resolve=0 success cases (same paths must OPEN), and EXDEV needs per-mount st_dev which carrick lacks (hardcoded st_dev:1) — that's namespace-aware path-walking, not "struct-translation/medium". Also omits the mode EINVAL checks openat203 actually tests (nonzero mode w/o O_CREAT, mode bits outside 07777); open_at_path silently masks mode &0o7777.

**#6 Cheap missing-syscall stubs: ioprio_get/set, vhangup, get_robust_list, rt_tgsigqueueinfo** — SOUND: Confirm. All four nrs (30/31/58/100/240) are genuinely absent from the dispatch table (verified, fall through to ENOSYS — not jitter/under-enforcement); macOS lacks ioprio (Linux-only) and vhangup (XNU nosys), and the fix correctly emulates rather than host-forwards. Test counts verified (4+1+1+1=7). ioprio_set01/03 need exactly the proposed per-pid packed store (round-trip + value-unchanged-after-EINVAL); rt_tgsigqueueinfo01 is intra-process and maps cleanly onto the existing route_thread_signal/record_pending_siginfo machinery (already used by rt_sigqueueinfo). Two minor notes, not flaws: get_robust_list01 doesn't compare head to set-value but DOES require EPERM-on-PID-1 (needs a cred check via existing creds.euid, slightly more than a stub) and set_robust_list currently stores nothing so the "store head/len" prerequisite is real; ioprio must accept which=PROCESS with who=0 (self), not just which 1..3. Effort "small" is right.

**#7 set*id all-success-path + setfsuid/setfsgid previous-value state** — FLAW: setregid02 is misclassified: it is an UNPRIVILEGED all-EPERM test (caller=nobody, 6 cases all expect EPERM per LTP source; sweep log docker(12,0,0,0) vs carrick(0,12,0,0)), NOT an all-success/saved-id test. carrick fails it because setregid (creds.rs:181) unconditionally returns 0 — an under-enforcement bug needing EPERM gating, which neither the fsuid/fsgid bookkeeping NOR the rank-2 saved-id rule delivers. The "subsumed by rank 2" claim is wrong for it (likely confused with setresgid02). So 1 of the 7 claimed tests will not clear. The setfs core (setfsuid01/setfsgid01/02/setfsuid03) IS sound: pure in-memory CredState bookkeeping, no macOS primitive assumed, "return prev; apply if privileged or target in {r,e,s,fs}" correctly matches the LTP setfsuid03/setfsgid02 semantics; effort "small" is right; it correctly excludes the enforcement-requiring setfsuid04. The flaw is the cluster scoping/merge, not the setfs mechanism.

**#8 read/pread/readv/write/pwrite/pwritev special-fd + count==0 errno mapping** — FLAW: Partly right but mis-scoped/mis-sized. Sound parts: write02/pwrite03 count==0 IS a real bug (read_bytes(NULL=0,len=0) returns EFAULT because contains_range requires addr>=region.start, so write(fd,NULL,0) gives EFAULT not 0); readv directory→EISDIR (currently EINVAL, fs.rs:2653) and pread64 dir→EISDIR/pipe→ESPIPE (currently EINVAL, fs.rs:2715-2727) are real. Flaws: (1) pread02 also asserts negative-offset→EINVAL, which the proposal omits and pread64 does NOT handle (usize::try_from accepts it, in-mem path returns 0) — read02/readv02 also assert O_DIRECT-unaligned→EINVAL (accepted as no-op; macOS has no O_DIRECT) — so those tests can't fully pass. (2) pwritev2 "negative pos→EINVAL" is semantically WRONG: pwritev201 requires offset==-1 to mean 'use current position', and pwritev202 wants invalid-flag→EOPNOTSUPP; the proposal conflates v1 (-1=EINVAL) with v2 (-1=current offset). (3) preadv2/pwritev2 (286/287) are SupportLevel::Deferred with NO dispatch arm = ENOSYS, so 'align + RWF_* validation' means writing two new handlers, not a small errno tweak — effort 'small' is underestimated. (4) Wording 'mirror pipe/socket→ESPIPE into read' would regress plain read() on pipes/sockets (must succeed); only positional variants get ESPIPE. Root cause (under-enforcement, not Docker jitter) is correctly classified.

**#9 POSIX AIO family (io_setup/submit/getevents/destroy/cancel) synchronous impl** — FLAW: Root cause + missing-syscall classification are correct and the in-process synchronous model is architecturally sound, BUT the proposal's io_cancel semantics are wrong: it says "io_cancel returns EINVAL" while the only io_cancel test in the cluster (io_cancel01) passes a zeroed ctx + NULL iocb and expects EFAULT (kernel does get_user(&iocb->aio_key) before the ctx lookup) — implemented as described it fails that test.

**#10 Small high-confidence errno fixes: pidfd CLOEXEC, getrandom flags, capget pid, fadvise advice/ESPIPE, ftruncate EINVAL, fdatasync EINVAL** — SOUND: CONFIRM.
